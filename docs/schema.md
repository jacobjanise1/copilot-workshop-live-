# Task Manager CLI — Technical Schema

## 1. Data Models

### Task
A task represents a unit of work with metadata for tracking and prioritization.

| Property | Type | Required | Validation |
|----------|------|----------|-----------|
| `id` | `string` | Yes | Must be a valid UUID from `crypto.randomUUID()` |
| `title` | `string` | Yes | Non-empty, max 255 characters |
| `description` | `string` | No | Default: empty string; max 1024 characters |
| `status` | `string` | Yes | Must be one of: `todo`, `in-progress`, `done` |
| `priority` | `string` | Yes | Must be one of: `low`, `medium`, `high`; default: `medium` |
| `createdAt` | `string` | Yes | ISO 8601 timestamp; set at creation, never modified |
| `updatedAt` | `string` | Yes | ISO 8601 timestamp; set at creation, refreshed on every update |

### Validation Rules

Each property has specific validation constraints applied before storage:

#### `id`
- Must be a valid UUID v4 generated by `crypto.randomUUID()`
- Format: 36 characters (8-4-4-4-12 with hyphens)
- Set automatically at task creation; never modified by user or API
- Cannot be null or undefined

#### `title`
- Required field; cannot be empty or null
- Must be a non-empty string after trimming whitespace
- Maximum length: 255 characters
- Trimmed of leading and trailing whitespace before storage
- Disallowed: null, undefined, or whitespace-only strings

#### `description`
- Optional field; defaults to empty string if not provided
- Accepts empty string, null (converted to empty), or any string up to 1024 characters
- Maximum length: 1024 characters (enforced after trimming)
- Trimmed of leading and trailing whitespace before storage
- Whitespace-only values converted to empty string

#### `status`
- Required field; must be one of the predefined enum values
- Valid values: `"todo"`, `"in-progress"`, `"done"`
- Case-sensitive (lowercase only)
- Defaults to `"todo"` if not provided on creation
- Cannot be null or undefined

#### `priority`
- Required field; must be one of the predefined enum values
- Valid values: `"low"`, `"medium"`, `"high"`
- Case-sensitive (lowercase only)
- Defaults to `"medium"` if not provided on creation
- Cannot be null or undefined

#### `createdAt` & `updatedAt`
- Both required; ISO 8601 format (e.g., `2024-03-31T14:30:00Z`)
- Generated automatically using `new Date().toISOString()`
- `createdAt` is set once at task creation and never modified
- `updatedAt` is set at creation and updated on every task modification
- Cannot be null, undefined, or manually set by user
- Must represent a valid date; invalid dates cause validation failure

### Constants
Shared enumerations and sort mappings used throughout the app.

| Constant | Type | Value |
|----------|------|-------|
| `VALID_STATUSES` | `string[]` | `["todo", "in-progress", "done"]` |
| `VALID_PRIORITIES` | `string[]` | `["low", "medium", "high"]` |
| `PRIORITY_ORDER` | `Record<string, number>` | `{ high: 0, medium: 1, low: 2 }` |
| `DEFAULT_STATUS` | `string` | `"todo"` |
| `DEFAULT_PRIORITY` | `string` | `"medium"` |

## 2. File Structure

```
src/
├── index.js              Entry point; parses CLI arguments and routes commands
├── task-store.js         In-memory Task[] array and raw CRUD operations
├── task-service.js       Business logic; wraps task-store with validation
├── validators.js         Input validation and enum checking utilities
├── formatters.js         Console output formatting and table rendering
├── constants.js          Enumerations and shared constants
└── types.js              JSDoc type definitions (Task, ListOptions, etc.)
```

## 3. Module Responsibilities

### `constants.js`
**Exports:**
- `VALID_STATUSES`, `VALID_PRIORITIES`, `PRIORITY_ORDER` (arrays and maps)
- `DEFAULT_STATUS`, `DEFAULT_PRIORITY` (default enum values)

**Dependencies:** None

**Purpose:** Single source of truth for all enum values and priority ordering logic.

---

### `types.js`
**Exports:**
- JSDoc `typedef` for `Task` shape
- JSDoc `typedef` for `ListOptions` (filters, sort key, sort direction)
- JSDoc `typedef` for operation results

**Dependencies:** None

**Purpose:** Shared type annotations for IDE autocomplete and documentation.

---

### `validators.js`
**Exports:**
- `isValidTitle(title: string): boolean`
- `isValidDescription(description: string): boolean`
- `isValidStatus(status: string): boolean`
- `isValidPriority(priority: string): boolean`
- `isValidTaskUpdate(update: object): { valid: boolean, errors: string[] }`

**Dependencies:** `constants.js`

**Purpose:** Centralized field and enum validation; returns boolean or detailed error list.

---

### `task-store.js`
**Exports:**
- `getTasks(): Task[]` — returns shallow copy of all tasks
- `createTask(task: Partial<Task>): Task` — stores new task, returns with id and timestamps
- `updateTaskById(id: string, update: Partial<Task>): Task | null` — returns updated task or null if not found
- `deleteTaskById(id: string): boolean` — removes task, returns true if found and deleted
- `getTaskById(id: string): Task | null` — returns a single task by id or null

**Dependencies:** `constants.js`, `types.js`

**Purpose:** Low-level in-memory storage (task array). No validation; assumes data is pre-validated.

---

### `task-service.js`
**Exports:**
- `createTask(input: { title: string, description?: string, status?: string, priority?: string }): { success: boolean, data?: Task, error?: string }`
- `listTasks(options?: ListOptions): Task[]` — returns filtered and sorted tasks
- `updateTask(id: string, update: Partial<Task>): { success: boolean, data?: Task, error?: string }`
- `deleteTask(id: string): { success: boolean, error?: string }`

**Dependencies:** `task-store.js`, `validators.js`, `constants.js`

**Purpose:** Business logic layer. Validates all inputs, applies defaults, manages timestamps, calls store, and returns standardized result objects.

---

### `formatters.js`
**Exports:**
- `formatTaskTable(tasks: Task[]): string` — returns ASCII table with columns: id, title, status, priority, createdAt, updatedAt
- `formatEmptyState(): string` — returns user-friendly message when no tasks exist
- `formatError(message: string): string` — formats error output for console

**Dependencies:** None (pure formatting functions)

**Purpose:** All console output formatting; keeps presentation logic separate from business logic.

---

### `index.js`
**Exports:** None (entry point)

**Imports:** `task-service.js`, `formatters.js`

**Responsibilities:**
- Parse command-line arguments (e.g., `node index.js create --title="Buy milk"`)
- Route commands to corresponding service methods
- Handle result objects and call formatters
- Exit with appropriate code on error (non-zero) or success (0)

**Supported commands:**
- `create [--title] [--description] [--status] [--priority]`
- `list [--status] [--priority] [--sort] [--order]`
- `update <id> [--title] [--description] [--status] [--priority]`
- `delete <id>`

## 4. Error Handling Strategy

### Error Categories

#### Validation Errors
**When thrown:** Input validation fails (invalid enum, empty title, non-existent id, malformed argument).  
**Where:** `task-service.js` methods before store operations.  
**Example:** `{ success: false, error: "Invalid status: xyz" }`

#### Not Found Errors
**When thrown:** Update or delete called with non-existent task id.  
**Where:** `task-service.js` updateTask and deleteTask.  
**Example:** `{ success: false, error: "Task not found: 123" }`

#### CLI Argument Errors
**When thrown:** Command syntax is wrong or required argument is missing.  
**Where:** `index.js` argument parsing.  
**Example:** `Task ID is required for delete command` (printed to stderr, exit code 1)

### Error Message Format
- Service-layer errors: `{ success: false, error: "<type>: <detail>" }`
- CLI errors: Print to stderr and exit with code 1
- All error messages are human-readable and suggest remediation where applicable

### Exit Codes
- `0` — Successful command execution
- `1` — Validation error, not found, or invalid arguments
- `2` — Unexpected runtime error (e.g., JSON parse failure)
